# CONTEXT: cribl-app (Cloud shell)

Purpose: the Cribl App Platform target - a Vite/React SPA packaged as a .tgz and installed into a Cribl.Cloud leader, running in a sandboxed iframe.

Platform rules live in AGENTS.md (read it before touching fetch, navigation, or config): locked fetch with automatic auth, app-scoped KV store (encrypted entries are write-only from the client), proxies.yml external-domain declarations with server-side header injection, policies.yml product API grants, 30s proxy timeout, 100 req/min.

Boundaries: binds platform adapters to @soc/core ports and renders @soc/ui. App-specific code here should shrink toward wiring-only as core/ui fill in.

Key files: config/proxies.yml and config/policies.yml (admin-reviewed external surface - update in the same PR as the feature needing them), scripts/package.mjs (version bump + .tgz build), vite.config.ts (dev-mode config watcher + package endpoint).

Status: shared app frame wired in (porting-plan Unit 1). App.tsx now flows AuaGate -> ModeSelect -> AppFrame: acceptance and mode persist as PLAIN KV entries (auaAcceptance / appMode) through PlatformSecretsStore, parsed by the tolerant @soc/core codecs; resolveFramePhase keeps the gate from flashing for accepted users. The frame's routes are Onboard (requires both live sides), Spike Harness (diagnostics, always available, panels unchanged), and Settings (platform info incl. the consolidated poll's platform-link status, current mode + Reconfigure = write EMPTY_MODE_RECORD + reload, and the validate-before-save JSON editor over the active connection's non-secret config). The old view-switcher tabs are gone; the connection bar stays visible inside the frame above content. useConsolidatedPolling registers the single connection-status poll (KV key listing, 60s, budget-capped) - every future poll registers there, never its own setInterval. Onboard still builds makeCloudPorts(activeTenant) and mounts @soc/ui's OnboardTableScreen inside PortsProvider, gated behind the five-field completeness check; its incomplete-connection guidance now points scope selection at the Azure Targeting route (identity still comes from harness panel 3). Unit 2: the Azure Targeting route (requires: azure) mounts @soc/ui's AzureTargetingScreen; the explicit Use this target commit runs @soc/core commitTargetScope over the profile store (merge, never replace), applies the invalidation (scope changes only stale permission results; token/secret flags honored defensively), and surfaces the consequence through the connection bar's switchNotice, next to the new committed-scope chip (formatScopeChip). PlatformAzureManagement gained requestUrl for ARM nextLink pagination, HARD-restricted to https://management.azure.com/ before any fetch (same token flow + 401-retry-once; management.azure.com was already declared in proxies.yml). Panel-4 discovery in the harness stays as diagnostics - the targeting screen is the product path. Feature implementation follows the roadmap in ../../docs/feature-catalog.md.
