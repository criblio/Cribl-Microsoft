# CONTEXT: @soc/local-app

Purpose: the local deployment target for customer-managed (on-prem) Cribl - a Node host that serves the shared UI in a browser and fulfills the same @soc/core port contracts the cloud platform provides to cribl-app. Launched from source (EDR-friendly, per the repo's established launcher pattern).

## Status

Host implemented (src/host/, plain ESM .mjs with JSDoc types, ZERO runtime dependencies - node builtins and global fetch only). The Cribl leader login flow is now IMPLEMENTED for cloud (login.cribl.cloud client credentials) and onprem (/api/v1/auth/login username/password) via the auth manager in src/host/cribl-auth.mjs; a static token remains supported as auth type "token". Web side implemented too: src/web/ is a Vite+React entry (main.tsx -> local-app.tsx) that imports "@soc/ui/styles.css" plus local.css chrome and now mounts the SHARED APP FRAME (porting-plan Unit 1): AuaGate -> ModeSelect -> AppFrame, with acceptance/mode persisted as PLAIN entries (auaAcceptance / appMode) through the local SecretsStore adapter and parsed by the tolerant @soc/core codecs. Frame routes: Onboard (requires both live sides; the shared OnboardTableScreen inside PortsProvider bound to the six local adapters in src/web/local-adapters.ts - AzureManagement -> POST /api/azure/request, CriblClient -> /api/cribl/*, SecretsStore -> /api/secrets* with write-only encrypted parity, JobStore -> /api/jobs*, UserContext -> /api/user, ArtifactSink -> Blob+anchor download; all requests ride a race-based fetchWithTimeout for cross-shell consistency) and Settings (the old header text + config summary graduated into its platform info rows; Reconfigure writes EMPTY_MODE_RECORD then reloads; no JSON editor here - the config is file-managed, and the note says where). useConsolidatedPolling registers the single connection-status poll (loopback GET /api/config, 30s). Unit 2: the Azure Targeting route (requires: azure) mounts the shared AzureTargetingScreen; Use this target persists the three scope fields as the PLAIN azureTargetScope secrets entry (tolerant parseTargetScope codec), merged over the file config's scope on load - identity always stays in config/local-config.json - with the invalidation consequence computed by @soc/core computeInvalidation; the committed scope renders as the frame topBar chip. LocalAzureManagement gained requestUrl -> POST /api/azure/request-url, the host's ARM full-URL proxy for nextLink pagination: the URL must start with https://management.azure.com/ (validated at the route AND again in azure.mjs before the bearer is attached - SSRF hard reject), same bounded timeout and 401-retry-once as /api/azure/request. Build: "npm run build" (tsc -b && vite build) emits dist/web, which the host serves at "/"; "npm run dev" proxies /api to http://127.0.0.1:4600; the root "npm run local" builds then starts the host.

## What the host provides

- Config: config/local-config.json (gitignored; copy config/local-config.example.json). Validated at startup with actionable per-field errors. Holds the Azure service principal (including clientSecret) and the Cribl leader URL + auth block { type: "cloud" | "onprem" | "token", ...per-type fields } with per-type required-field validation. The legacy flat cribl.authToken is still accepted as type "token" with a deprecation note in the startup log.
- GET /api/config: the NON-secret Azure fields (AzureConfig shape, setupPath "existing") plus criblLeaderUrl for display. Secrets never appear on any HTTP response.
- POST /api/azure/request: ARM proxy. The host acquires and caches the client_credentials token (refresh ~5 min before expiry; on upstream 401 re-acquire once and retry once) and returns the upstream {status, body} verbatim - twin of the cloud shell's PlatformAzureManagement.
- POST /api/cribl/request and GET /api/cribl/groups: leader proxy whose bearer comes from the Cribl auth manager (src/host/cribl-auth.mjs). Auth type "cloud" mints via the login.cribl.cloud client-credentials flow (JSON body, audience https://api.cribl.cloud, cached until ~5 min before expiry); "onprem" logs in via {leaderUrl}/api/v1/auth/login (response carries no expires_in - lifetime assumed 3600s, cached); "token" passes the static configured token through. On a leader 401 with a mintable type the proxy re-authenticates ONCE and retries ONCE (twin of the Azure proxy's 401 recovery); with a static token the 401 stays data and the host logs a tokens-expire hint. Auth failures name the exact cribl.auth field to fix (cloud 401/403 -> invalid client id or secret; on-prem 401 -> invalid username or password; on-prem 429 -> rate limited with retry-after; DNS/connect -> check leaderUrl; TLS -> rejectUnauthorized guidance). Groups mapping mirrors the cloud adapter's tolerant shape-handling. cribl.rejectUnauthorized=false (self-signed on-prem leaders) applies via an https.Agent to leader calls ONLY - including the on-prem login; the cloud token mint always verifies TLS.
- Secrets (data/secrets.json, 0600 best effort): PUT/GET/DELETE /api/secrets/{key}, POST /api/secrets-list. Values stored with { encrypted: true } are WRITE-ONLY through the API - GET returns { value: null }, exactly like the cloud KV's unreadable encrypted entries (SECURITY POSTURE PARITY).
- Jobs (data/jobs.json): POST/GET /api/jobs, GET/PATCH /api/jobs/{id}. Same contract as the cloud PlatformJobStore: store-managed id/createdAt, patch-wins shallow merge, updatedAt refreshed, newest-first listing. Atomic-ish writes (temp file + rename).
- GET /api/user: { id, username } from the OS account (node:os userInfo).
- Static serving of dist/web with index.html fallback for client routing; friendly "run the build" page when the build output is missing.

Every upstream request (Azure token, ARM, leader) is bounded by an ~30s AbortController timeout. Node's fetch honors AbortSignal - the ignore-abort quirk that forces Promise.race timeouts exists only in the Cribl platform's locked fetch bridge on the cloud side.

## Security posture

- Binds 127.0.0.1 only. There is NO API auth: this is a single-operator localhost tool and the loopback interface is the boundary. Do not port-forward or reverse-proxy it.
- The Azure client secret is read ONLY from config/local-config.json and used server-side for token acquisition; it is never exposed over HTTP. Same for the Cribl credentials (auth.clientSecret / auth.password / auth.token): minted leader tokens live only in process memory and never appear in any HTTP response or log line.
- Encrypted-marked secrets are unreadable through the browser-facing API (write-only parity with the cloud KV).

## Deliberately deferred

- Cribl.Cloud management plane (gateway.cribl.cloud) and multi-workspace enumeration: leaderUrl pins ONE workspace host (the 'main-{orgId}' pattern); the minted cloud bearer would work against other workspace hosts and the gateway, but nothing calls them.
- 429 backoff beyond surfacing the login retry-after in the error message (no automatic wait-and-retry).
- Encryption at rest for data/secrets.json: the encrypted marker enforces API write-only semantics, but the file itself is plaintext on disk (0600 best effort; mode bits are advisory on Windows). data/ is gitignored.
- API auth on the loopback listener (documented above as out of scope for a single-operator tool).
- First-run onboarding GUI covering both targets, drift-check job scheduling, outbound domain allowlist enforcement (mirror of cloud proxies.yml discipline) - these arrive with the web UI and later phases.

Advantages over the cloud target (capability superset): private network reach, no 30s/100rpm proxy limits, scheduling, can run inside air-gapped networks beside a local leader.

## Unit 3: logging and in-app diagnostics

Browser side: src/web/logger.ts HostLogger implements the @soc/core Logger port by batching entries (adapter-stamped timestamps) to POST /api/logs - one fire-and-forget POST per 20 entries or 3 seconds, dropped on failure by design. makeLocalPorts takes the page-lifetime instance (LocalPorts now carries logger - usecases log for free), and local-app.tsx registers the Logs route (requires 'none'): @soc/ui LogsScreen over GET /api/logs?tail=500, re-parsed through the shared parseLogLine (the pinned core line format round-trips), with the support bundle downloaded via the ArtifactSink port.

Host side: src/host/logger.mjs is the file logger - appends formatLogLine-format lines (a documented runtime twin of @soc/core log-model; keep in lockstep) to data/logs/app.log with a 10MB cap and a SINGLE app.log.1 rollover (legacy Electron logger mined as rotation prior art, reimplemented), order-preserving serialized writes that never throw, warn/error mirrored to the console, and a tail() that spans the rollover. POST /api/logs sanitizes every shipped entry server-side (sanitizeLogEntries: batch cap 100, message/context caps, non-primitive context values DROPPED - the no-secrets hard rule enforced at the loopback boundary too); GET /api/logs?tail=N (default 500, max 2000) returns { lines }. The host's own events log through the same file: one line per finished API exchange (method/path/status/ms; /api/logs itself and static requests excluded; bodies never logged), ARM token acquisitions and 401-retry recoveries (azure.mjs), and Cribl cloud mint / on-prem login successes and failures (cribl-auth.mjs) - metadata and curated error text only, never a token, secret, or password value.

## Unit 4: deployment and naming options

The Options route (requires 'none') mounts @soc/ui OptionsScreen over ONE plain host-secrets entry appOptions (same key name as the cloud shell's KV entry), following the appMode persistence pattern: local-app.tsx hydrates the parsed options alongside acceptance/mode/scope on mount and passes loadOptions/saveOptions callbacks over ports.secrets; saves merge through @soc/core applyOptionsPatch (unmanaged blob keys survive) and refresh the shell's parsed copy, so the Onboard route's criblDefaults (worker-group preselect + destination-id prefix/suffix) follow immediately. No new host surface: the existing /api/secrets routes carry it.

## Unit 6: batch onboarding

The Batch Onboard route (requires 'both') mounts @soc/ui BatchDeployScreen against the same six local adapters, gated on the host config load like Onboard. The shell injects the module-scoped BATCH_PACING hooks (Date.now/setTimeout) for the usecase's rolling-minute ARM budget (loopback has no proxy budget, but ARM is rate-limited upstream, so the @soc/core default 80 req/min stays) plus the persisted appOptions, and hands nav.navigate('options') as the open-Options hook. templateOnly artifacts download through the existing ArtifactSink adapter (Blob + anchor). No new host surface: the batch rides the existing /api/azure/request, /api/cribl/*, and /api/jobs* routes.
