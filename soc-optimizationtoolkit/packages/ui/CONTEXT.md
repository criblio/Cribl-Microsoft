# CONTEXT: @soc/ui

Purpose: the shared React feature screens and components rendered by both shells (cribl-app iframe and local-app browser UI).

Boundaries: imports only @soc/core. No direct fetch or storage access; all IO flows through core ports provided via React context by the hosting shell.

Layout convention: one folder per feature domain from the catalog (onboarding, dcr, packs, discovery, governance, lookups, migration, drift, labs) plus frame/ (app chrome + gates), screens/ (settings), components/ (shared widgets), polling/ (the one budgeted poller).

Status: shared app frame + settings live (porting-plan Unit 1), joining the walking-skeleton onboarding screen. src/ports-context.ts defines PortsContext / PortsProvider / usePorts, carrying the six @soc/core port instances plus the active non-secret AzureConfig; onboarding/OnboardTableScreen drives the @soc/core onboardTable use-case (worker-group picker via cribl.listGroups, transient write-only-aware secret input, live step list from onProgress, honest summary and role-assignment guidance). frame/: AppFrame (route-table sidebar filtered by @soc/core filterNavItems, mode chip, topBar shell-chrome slot, content area), AuaGate (acceptance gate with rewritten platform AUA text, scroll-to-bottom enable, unscrollable-content escape), ModeSelect (four-mode first-run chooser), and frame-state.ts - the pure, tested layer: resolveFramePhase (loading contract: the gate NEVER flashes for an already-accepted user), EMPTY_MODE_RECORD (the Reconfigure-writes-empty-object contract), MODE_LABELS/MODE_OPTIONS (one label map, one chooser list), isScrolledToBottom. screens/: SettingsScreen (shell-provided platform info rows props bag, current mode + Reconfigure, and the validate-before-save raw-JSON editor over the active connection's non-secret config - optional, since the local shell's config is file-managed) with the pure validateConfigJson glue (parseAzureConfig underneath; refuses non-objects, surfaces dropped/coerced keys as warnings, never stores a pasted secret). components/InfoTip renders embedded newlines (pre-wrap) on hover/focus/click. polling/useConsolidatedPolling wraps the @soc/core poll-scheduler: the hook owns Date/timers, core owns the pure budget decisions; ALL status polls register here (only connection-status for now). screens/azure-targeting/ (porting-plan Unit 2): AzureTargetingScreen - the subscription -> workspace -> resource-group cascade over the @soc/core azure-discovery usecases with always-visible selectors (disabled-with-instructions before data), ONE loader effect fed by the pure buildLoaderPlan (the legacy page had three overlapping loaders), create-RG (legacy name-sanitization rule mined verbatim into sanitizeResourceGroupName) / create-workspace / enable-Sentinel actions with honest line-by-line output, an offline free-text branch (shell derives it from the frame mode), and browse-vs-commit separation: only the explicit "Use this target" button calls the shell's onCommitScope, whose invalidation consequence renders via commitNoticeText (connection-bar notice pattern); targeting-state.ts also exports formatScopeChip (the committed-scope topBar chip both shells render) and the tolerant parseTargetScope/serializeTargetScope codec the local shell persists its scope override with. Tests are pure and colocated (frame-state, config-json, step-line, targeting-state); DOM tests remain out of scope. Styling: src/styles.css is the single source of the class conventions (reset/body, panel, field, run-button, status, result, app-frame/mode-chip, gate/aua/mode-option, settings/config-editor, info-tip), exported as "@soc/ui/styles.css"; each shell imports it BEFORE its own shell-specific chrome stylesheet (cribl-app App.css, local-app local.css).
